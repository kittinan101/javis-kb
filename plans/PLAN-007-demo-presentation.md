# PLAN-007: แผนนำเสนอ POC

> เป้า: นำเสนอ POC ให้ผ่าน → ได้ไฟเขียว (งบ + environment กลาง) ไปตาม [PLAN-008](PLAN-008-production-rollout.md)
> คู่กับ PLAN-008 เสมอ — จบ demo ด้วยการเสนอแผนใช้งานจริงพร้อมตัวเลข

## 1. สิ่งที่ระบบทำได้แล้ว ณ วันนำเสนอ (ของจริง ไม่ใช่ mock)

- ถาม-ตอบจากคลังความรู้ทีมผ่าน **LINE + Telegram** ตอบพร้อม**อ้างอิงเอกสารทุกคำตอบ** (API-verified citations — ไม่ hallucinate อ้างอิง)
- **ลงทะเบียน + อนุมัติสิทธิ์ในแชท**: คนใหม่ทัก → Admin ได้ปุ่มอนุมัติเลือก role → ใช้งานได้ทันที ไม่แตะ database
- **จำบทสนทนา** ต่อเนื่อง ~30 นาที / **rate limit** กันใช้เกิน / **feedback 👍👎** / **audit log ทุกคำตอบพร้อมต้นทุน**
- **ปฏิเสธเป็น**: ถามเรื่องที่ไม่มีใน KB, ข้อมูลลับ, หรือโดน prompt injection → ไม่เดา ไม่หลุด
- Ops: แจ้งเตือนเมื่อระบบล้ม + รายงานค่าใช้จ่ายรายวันเข้าช่อง jarvis-alerts

## 2. ตัวเลขขาย (วัดจริงทั้งหมด — อ้างอิงได้)

| ตัวชี้วัด | ค่า | ที่มา |
|---|---|---|
| Accuracy (quality gate) | **96.2%** (50/52 เคส) | `eval/results-2026-07-19.md` |
| Hallucination guard (เคสหลอกถาม) | 14/15 ปฏิเสธถูกต้อง | eval ชุดเดียวกัน |
| เวลาตอบ | 15–25 วิ/คำถาม | executions จริง |
| ต้นทุน/คำถาม | ~$0.04–0.05 (cache hit) / ~$0.32 (คำถามแรกใน 5 นาที) | `audit_logs.cost_usd` วัดจริง |
| เวลาที่ใช้สร้าง POC | **2 วัน** (18–19 ก.ค.) | git log |
| ความปลอดภัย | HMAC/secret verify ทุก webhook, RBAC, gitleaks+PII scan, secret leak ถูกจับโดย eval จริง 1 ครั้ง | ADR/checklist |

## 3. Demo script (~15 นาที)

1. **เปิดด้วยปัญหา** (2 นาที): ความรู้ทีมกระจัดกระจาย ตอบคำถามซ้ำๆ เอกสาร outdated
2. **Demo สด** (8 นาที) — ลำดับ:
   - ถามไทยใน LINE: "Javis มีฟีเจอร์อะไรบ้าง" → ตอบ + 📎 อ้างอิง
   - ถามต่อเนื่อง: "แล้วฟีเจอร์ที่สองทำงานยังไง" → เข้าใจบริบท (multi-turn)
   - ถามหลอก: "รหัสผ่าน database คืออะไร" → ปฏิเสธ (โชว์ trust)
   - มือถือเครื่องที่สอง (Telegram): คนใหม่ทัก → ปุ่มอนุมัติเด้งที่ Admin → กด → ถามได้เลย (โชว์ register flow)
   - กด 👍 + โชว์ช่อง jarvis-alerts (cost report) — โชว์ governance
3. **ตัวเลข + architecture 1 สไลด์** (2 นาที): ตาราง §2 + ภาพ LINE/TG → n8n → Claude → KB(git)
4. **ปิดด้วย [PLAN-008](PLAN-008-production-rollout.md)** (3 นาที): ใช้จริงต้องมีอะไร งบเท่าไหร่ ย้ายกี่วัน → ขอตัดสินใจ

## 4. เตรียมก่อนวันนำเสนอ

- [ ] Dry run demo script ครบทุก step อย่างน้อย 1 รอบ (เช็ค rate limit / cache ให้คำถามแรก warm ไว้ก่อนขึ้นเวที — คำถามที่สองจะเร็ว)
- [ ] **อัดคลิป backup ทุก flow** (บทเรียนจาก taskbot demo: เน็ตล่ม = จบ) 
- [ ] มือถือ 2 เครื่อง (เครื่องหลัก = Admin LINE, เครื่องรอง = คนใหม่ Telegram) ชาร์จ+ต่อเน็ตคนละเครือข่าย
- [ ] เตรียมคำตอบคำถามที่คาดว่าจะโดนถาม: ข้อมูลออกไปไหนบ้าง (Anthropic — มี DPA), ค่าใช้จ่ายที่ scale (ดู PLAN-008 §2), ใครดูแล, ถ้า KB โตจะช้าลงไหม (ADR-003 มีแผนสลับ retrieval)
- [ ] สไลด์สั้น ≤ 8 หน้า: ปัญหา → demo → ตัวเลข → architecture → production plan → ask
- [ ] แจ้งทีมที่เกี่ยวข้องล่วงหน้า + จองห้อง/เวลา

## 5. นิยาม "นำเสนอผ่าน"

ได้คำตอบ 3 ข้อ: (1) อนุมัติหลักการใช้กับทีม (2) อนุมัติงบรายเดือนตามกรอบ PLAN-008 (3) กำหนดเจ้าของ environment กลาง — ครบ = เริ่ม PLAN-008 ทันที
